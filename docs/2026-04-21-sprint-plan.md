# Design: 2-Week Sprint — Foundation + One Real Teen

## Problem Statement

DriveMaster has 3,400+ questions, a mascot system, XP/streak/league engines, and most of a PWA. It also has a broken core loop, unauthenticated endpoints leaking answers, pervasive DESIGN.md violations, zero analytics, and zero real teen usage. The founder has a 12-feature wishlist and 2 weeks of solo effort. Wrong question: "which feature first?" Right question: "what gets one real Illinois teen retained enough that behavior, not guesswork, ranks the wishlist?"

## Demand Evidence

- **Today:** pre-users / waitlist stage. Landing page claims "10,000+ drivers, 95% pass rate, 4.9 rating" — unverified, legal risk, not real demand.
- **Status quo:** Illinois teens grind free IL Secretary of State practice tests (or driving-tests.org) the night before. Free, boring, emotionally disengaged.
- **Favorable:** switching cost from status quo is $0. DriveMaster wins on "not boring + not last-minute," not on feature count. Zutobi already out-features us.
- **Gap:** founder persona is self-modeled, not observed. No named teen currently using the app weekly.

## Status Quo

- Free IL SoS official practice quizzes (ugly, last-minute grind).
- driving-tests.org and similar ad-supported sites.
- Handbook PDF or nothing.
- Mandatory driver's ed class provides some coverage.

Workaround cost: low dollar, high emotional friction (boring, generic, last-minute panic).

## Target User & Narrowest Wedge

**User:** one specific Illinois resident preparing for the permit test, recruited by end of week 1. Could be a first-timer (15+), an adult new to Illinois, or a retaker — the observation goal is the same. Candidate sources: friend-of-family, local high-school driver's-ed instructor referral, r/Illinois or r/teenagers post, Illinois-focused Discord.

**Wedge:** a stabilized, instrumented, on-brand core loop (sign up → onboarding → quiz → results → Dash hype) that one real teen can run end-to-end without hitting broken nav, placeholder images, or fake-zero results screens. Retention feature selection deferred to week 2 based on observed behavior.

## Constraints

- Solo dev, ~2 weeks.
- Minors will be among users — elevated security + privacy bar applies (COPPA-adjacent, IL BIPA). Adult users don't reduce this bar.
- Must honor DESIGN.md (Cabinet Grotesk / DM Sans / JetBrains Mono; Blue/Orange/Green palette; 44px touch targets; Dash tone rules).

## Premises

1. The 12-feature wishlist is premature optimization until one real teen retains.
2. Core loop bugs (nav shell returns null, quiz images don't render, results page fakes zeros) are stop-the-line — no feature ships on top of them.
3. Shipping without analytics teaches you nothing; week 1 must include instrumentation.
4. Security gaps involving minors are reputational, not polish. Must close in week 1.
5. Purple violations are most lethal on the activation surface (onboarding + landing), not the dashboard. Prioritize purge by funnel position.
6. The recommended retention feature for week 2 will be picked from observed behavior, not founder intuition.

## Cross-Model Perspective

Not run this session.

## Approaches Considered

### Approach A: Feature Sprint
Ship push notifications + daily quests + share cards on top of current app. Looks productive, compounds regressions, produces zero validation. Rejected.

### Approach B: Foundation + One Teen (selected)
Week 1 stabilize and instrument. Week 2 recruit one real teen, observe, ship one targeted retention feature. See Recommended Approach below.

### Approach C: Dogfood First
No new code for 2 weeks. Put the broken app in front of 5 teens, record Clarity sessions, rewrite the wishlist from observed behavior. Highest-learning per hour, but leaves critical security and core-loop bugs in place — unacceptable given minor users.

## Recommended Approach

**Approach B — Foundation + One Teen.**

### Week 1 — Foundation (Monday → Sunday)

**Day 1-2: Stop-the-line fixes**
- Restore nav shell in `components/navigation.tsx` and wire `MobileNavigation` / `DesktopNavigation` into `components/mobile-layout.tsx`.
- Render actual quiz images in `components/quiz/question-card.tsx:88-94` (replace the `ImageIcon` placeholder with `<img>` or `next/image`).
- Fix results-page zero-fallback in `app/(quiz)/quiz/results/page.tsx:29-48` — redirect to `/quiz` with a toast if sessionStorage is missing, never render a fabricated failure.

**Day 2-3: Critical security gaps**
- Add auth to `/api/questions` GET, `/api/questions/random` GET, `/api/questions/seed` GET, `/api/explain` POST, `/api/challenges/[id]` GET.
- Strip `correct_answer` + `explanation` from any unauthenticated response path that must remain public.
- Rotate `CRON_SECRET` and `ADMIN_API_KEY` to cryptographically random values, add to `scripts/validate-env.ts`.
- Add `@upstash/ratelimit` + Vercel KV on `/api/parent`, `/api/explain`, `/api/leaderboard/update`, auth endpoints.
- Fix parent-link auto-approval: require teen-side confirmation before parent gains read access to progress data.
- Convert XP update in `app/api/leaderboard/update/route.ts:91-98` to atomic SQL `UPDATE ... SET weekly_xp = weekly_xp + $1`.

**Day 4: Analytics + consent**
- Install PostHog (free tier) with `session_recording.maskAllInputs: true`, no replay on auth routes, `person_profiles: 'identified_only'`, reverse-proxy via `/ingest`.
- Install Microsoft Clarity via `next/script` `afterInteractive`.
- Add Meta Pixel + TikTok Pixel stubs via `next/script` (load only after analytics consent).
- Ship an analytics consent toggle in profile; default OFF for under-16 where we can detect it.
- Update privacy policy to disclose PostHog + Clarity + Meta/TikTok sub-processors. Non-negotiable before any paid traffic.
- Instrument: `signup_started`, `signup_completed`, `onboarding_step_viewed`, `onboarding_completed`, `first_quiz_started`, `quiz_completed`, `question_answered {correct,time_ms}`, `quiz_abandoned {at_question}`, `streak_incremented`, `push_opt_in`.

**Day 5-6: DESIGN.md purge (activation-surface priority)**
- Replace every `purple-*` class in `app/(onboarding)/**`, `app/page.tsx`, `app/(dashboard)/dashboard/page.tsx:120`, and `components/onboarding/**` with `orange-*` or palette-token equivalents.
- Fix `#8B5CF6` / `#EC4899` in `lib/constants/index.ts:145,152` and `lib/data/questions/categories.ts:41,59`.
- Wrap Framer Motion components with `useReducedMotion()` (start with the quiz + onboarding paths; full codebase can wait).
- Raise `small` variant in `components/layout/touch-friendly.tsx:22` to 44px floor.
- Add Dash mascot to signup page and error page (`app/error.tsx`) — the two highest-emotion moments currently off-tone.

**Day 7: Recruit one Illinois permit-test studier**
- Post in r/Illinois, r/teenagers, and 2-3 Illinois-area high school Discords.
- Ask 2 friends-with-connections and 1 driving-school owner for one referral each.
- Target: one named Illinois resident actively preparing for the permit test, willing to screenshare for 20 min + use the app for 5 days. Age doesn't matter — the behavior observation goal is the same.

### Week 2 — One Teen (Monday → Sunday)

**Day 8-9: Watch and log**
- 20-min moderated session. You stay silent. They sign up, onboard, take one quiz, check results. Record via Clarity plus video.
- After the session, review: where did they hesitate? Where did they smile? What did they not tap that you thought they would? What did they ask about?

**Day 10: Pick ONE retention feature from observed behavior**
Priority order if behavior is ambiguous:
1. Streak-or-shame push notification (if they said anything resembling "I'll forget")
2. Daily quest UI (if they expressed "what do I do next?" friction)
3. Animated TikTok/Story-sized share card (if they asked "can I share this?")

Ship exactly one. Instrument it with an event. Do not ship two. The point of the 2-week cut is that behavior ranks the wishlist, not intuition.

**Day 11-13: Build + ship the one feature.**

**Day 14: Re-observe**
- Same teen, 10-min session. Did the feature change the behavior it was designed for? Yes → keep. No → remove.

## Open Questions

- Is push notification infra realistic solo in 2-3 days, or should we default to Resend email + SMS for streak reminders? (Answer depends on VAPID setup time; spike on day 7.)
- Do we need to pull the unverified "10,000+ users, 95% pass rate, 4.9 rating" landing-page claims before showing the app to any real teen? (Recommendation: yes, before day 8.)
- Spanish locale: next-intl is installed but unused. Defer until after this sprint.

## Success Criteria

- **Week 1:** core loop works end-to-end on mobile; all 6 critical auth gaps closed; admin secrets rotated; PostHog + Clarity live with consent; purple removed from all onboarding + landing + dashboard files; one named teen recruited.
- **Week 2:** one real Illinois teen has signed up, onboarded, completed at least 3 quizzes, and triggered the shipped retention feature. You have recorded video/Clarity sessions of both visits. The feature either demonstrably changed behavior or gets removed on day 14.

## The Assignment

**Before you touch a single line of code:** today, open your phone. Text two people — a friend, family member, or anyone with a connection to someone preparing for the Illinois permit test. One sentence: "Hey, any chance you know someone studying for the Illinois permit test who'd spend 20 minutes trying a driving app I'm building? I'll Venmo them $20." You cannot start week 1 without at least one yes or one lead in motion, because week 2 depends on the recruit and recruiting always takes longer than coding.

## What I noticed about how you think

- You came in with a 12-feature wishlist and asked me to force-rank it. Good instinct: you already knew more features was the wrong answer or you wouldn't have asked someone else to rank them.
- You picked "pre-users / waitlist" honestly on the demand question. Most founders pick "real users" and bend the definition. You didn't.
- You picked "self-modeled" on the persona question instead of claiming a teen you don't have. That self-awareness is the thing the 2-week plan is built to fix.
- You picked the boring foundation approach over the feature sprint without flinching. A lot of solo founders pick A because A looks like shipping. B is shipping. A is performing.
